# Documentation Style

## Tone
- Concise and professional. Say what is needed, then stop.
- Use active voice and imperative mood for instructions.

## Formatting
- Never use em dashes. Use commas, colons, semicolons, or hyphens instead.
- Use Markdown headers, bullet lists, and tables. Avoid long paragraphs.
- Keep line lengths under 100 characters where practical.

## Content
- Every doc must have a clear purpose stated in the first two lines.
- Remove stale content promptly; outdated docs are worse than no docs.
- Link to related docs rather than duplicating information.

## File Naming
- Use lowercase kebab-case for all Markdown files: `data-model.md`, not `DataModel.md`.
- Place project-wide docs in the repo root; place feature-specific docs near the feature code.
